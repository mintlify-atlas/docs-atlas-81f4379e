The most important thing to understand about these projects is that they aren't static documentation to be read and filed away. They're practical frameworks, protocols, and small systems shaped by real use — and the fastest way to get value from them is to put them in front of an AI assistant and work through them together. Reading a behavioral contract in isolation is like reading a recipe without a kitchen. The protocol only becomes useful when it's active inside a session.

## The Core Workflow

<Steps>
  <Step title="Pick a project that fits your problem">
    Start with the [Introduction](/introduction) to get an overview of all five categories. Don't try to read everything — identify one problem you're actually dealing with right now and find the project that targets it most directly.
  </Step>
  <Step title="Paste the GitHub link into your AI assistant">
    Copy the repository URL for the project you've chosen and paste it directly into your AI chat. Most modern AI assistants can read GitHub repository content and will give you a working summary without you having to explain the project yourself.
  </Step>
  <Step title="Ask the AI to explain it, show examples, and tell you if it fits">
    Don't just ask for a summary. Ask the AI to explain what problem the project solves, walk through practical examples of it in use, and — critically — tell you whether it actually fits the specific problem you're trying to solve. That last question surfaces assumptions early.
  </Step>
  <Step title="Adapt and integrate into your own workflow">
    Take what fits, discard what doesn't, and adjust the parts that are close but not quite right. These protocols are starting points, not rigid specifications. A version that works in your workflow is more valuable than a version followed precisely but incorrectly.
  </Step>
</Steps>

## Quick-Start Questions

These prompts are designed to be pasted directly into your AI assistant alongside a repository link. Each one pulls a different kind of useful information from the project.

General orientation — use this first with any project:

```text
What is this project for, how does it work, and what could I use it for in my own workflow?
```

Identifying edge cases and failure modes before you commit:

```text
What are the practical limits of this protocol?
```

Finding the fastest path into the codebase or documentation:

```text
Which files should I read first?
```

Testing fit against a specific situation before investing time:

```text
Does this fit my specific problem: [describe your problem]?
```

## Matching Protocol to Problem

Not sure which project to start with? These scenarios map common AI-assisted workflow problems to the tools most likely to help.

<Accordion title="My AI keeps losing track of context">
  Two projects address this directly. **Canon Boundary Guard** (available for both Codex and GPT Projects) keeps project evidence, chat context, operator instructions, working hypotheses, and model assumptions in separate cognitive layers so they don't silently merge into a single undifferentiated blob. [**Memory-Assisted Shaping**](/contracts/memory-assisted-shaping) is built specifically for long ChatGPT and GPT Project sessions where decisions, gates, discarded paths, and source boundaries need to stay intact until the final artifact is ready.
</Accordion>

<Accordion title="I need to stress-test a plan or idea">
  [**PA-PVP**](/protocols/pa-pvp) (Plutonium-like Adversarial Peer Validation Protocol) turns vague plans, artifacts, or stuck decisions into a clear verdict, one next action, and a continuation point — rather than another open-ended discussion. If you need something lighter, [**PA-PVP mini**](/protocols/pa-pvp-mini) is the trimmed version for quick adversarial review of ideas, prompts, procedures, code snippets, or AI outputs. It finds gaps, weak logic, missing constraints, and shaky fixes, then allows a second AI to challenge both the original work and the prior review.
</Accordion>

<Accordion title="My AI gives responses that are too shallow or too deep">
  [**A.D.A.M. (Adaptive Depth and Mode)**](/protocols/adam) is built for this problem. It helps keep AI responses calibrated to the actual complexity of what's being asked, so simple questions get simple answers and complex work doesn't collapse into an unclear or overloaded response. It's one of the most broadly applicable protocols in the hub.
</Accordion>

<Accordion title="Project decisions keep disappearing between sessions">
  [**Signal Rail**](/protocols/signal-rail) is the tool for this. It keeps live project material in the right place — decisions, open work, ideas, constraints, and handoff notes — so they don't collapse into one messy document between sessions. It's designed for projects that run across multiple chats and need a persistent structure that an AI can reliably navigate.
</Accordion>

<Accordion title="I want to start a structured ChatGPT Project">
  Two tools are useful here depending on where you are in the setup. [**GPT Project Forge**](/tooling/gpt-project-forge) (GPT-PF) prepares the starting files for a ChatGPT Project so scope, sources, constraints, and starting basis are clear before real work begins. Once the project is running, [**ChatGPT Skill System**](/tooling/chatgpt-skill-system) gives it a controlled session workflow for loading and using skill-like packages — boot the core, load named skills, and verify external skills before they enter context.
</Accordion>

<Tip>
  Adapt protocols rather than following them rigidly. A protocol that has been adjusted to fit your actual workflow is more useful than one applied by the letter in a context it wasn't designed for. The goal is a working system, not a correctly executed procedure.
</Tip>
